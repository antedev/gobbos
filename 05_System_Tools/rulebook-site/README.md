# Gobbos Rulebook Site — Developer Guide

This is a static-site generator that converts the game's Markdown rule files into a browsable HTML documentation site. It lives inside `05_System_Tools/` because it is a **tool** — it does not contain game rules itself.

> **Golden Rule:** This tool only *reads* from the rule directories. It never modifies any `.md` source files.

---

## Quick Reference

```bash
npm run build    # Generate the site into dist/
npm run dev      # Build + serve at localhost:3000 (live-reload)
npm run serve    # Serve dist/ without rebuilding
```

## Project Structure

```
rulebook-site/
│
├── build.js               ← The build script (all the logic lives here)
├── template.html           ← HTML shell for environment pages (stage, dev, prod)
├── index-template.html     ← HTML shell for the landing page
├── style.css               ← All styling (copied to dist/ on build)
├── package.json            ← Dependencies and npm scripts
│
└── dist/                   ← Generated output (do not edit by hand)
    ├── index.html
    ├── dev.html
    ├── stage.html
    ├── prod.html
    └── style.css
```

### What each file does

| File | Purpose |
|---|---|
| `build.js` | Scans the repo's rule directories, reads all `.md` files, parses them to HTML, generates sidebar navigation, and writes the final pages to `dist/`. |
| `template.html` | The HTML skeleton for content pages. Contains placeholders like `{{TITLE}}`, `{{NAV_CONTENT}}`, `{{MAIN_CONTENT}}` that the build script fills in. Also contains all client-side JavaScript (theme toggle, sidebar, scroll spy). |
| `index-template.html` | The landing page template. Only has one placeholder: `{{ENV_CARDS}}`. |
| `style.css` | Complete stylesheet with CSS custom properties for theming. Defines both light and dark mode. |

## How the Build Works

The build script follows this pipeline:

```
1. Scan directories     →  Find all .md files, sorted by numeric prefix
2. Build data model     →  Sections (folders) containing Pages (files)
3. Concatenate markdown →  One big markdown string per environment
4. Parse to HTML        →  markdown-it with anchor + wikilink plugins
5. Generate sidebar     →  HTML nav from the data model
6. Inject into template →  Replace {{placeholders}} with generated content
7. Write to dist/       →  One .html file per environment + index + CSS
```

### Content Discovery

The script scans three directories relative to the repo root:

| Directory | Output file | Description |
|---|---|---|
| `00_DEV_Brainstorms/` | `dev.html` | Flat structure — loose `.md` files become standalone sections |
| `01_STAGE_Drafts/` | `stage.html` | Nested structure — subfolders become collapsible sidebar groups |
| `02_PROD_Core_Rules/` | `prod.html` | Same as STAGE; currently empty (shows placeholder) |

### Sorting & Naming

Files and folders use numbered prefixes for ordering (e.g. `01_Characters & Mobs/10_Stats.md`).

- **Sorting:** The `numericSort()` function extracts the leading number and sorts numerically, so `02` comes before `10`.
- **Display names:** The `stripPrefix()` function removes the leading digits + underscore/space and the `.md` extension. So `01_Characters & Mobs` → `"Characters & Mobs"` and `10_Stats.md` → `"Stats"`.

### Anchor IDs (Slugs)

Every section and page gets a URL-friendly ID for sidebar linking:

- Folder section: `slugify("Characters & Mobs")` → `characters-and-mobs`
- File within folder: `characters-and-mobs--stats`
- Loose file: `enemy-stats-draft`

The `--` double-dash separates the folder slug from the file slug, ensuring uniqueness.

The `slugify()` function also converts `&` to `and` before stripping special characters, so `Characters & Mobs` doesn't become `characters--mobs`.

### Markdown Parsing

The parser is [markdown-it](https://github.com/markdown-it/markdown-it) with two plugins:

1. **markdown-it-anchor** — Automatically adds `id` attributes to all headings (`#`, `##`, etc.) based on the heading text. This is what makes wikilinks work.
2. **Custom wikilinks plugin** — Converts Obsidian-style `[[Page Name]]` syntax into `<a href="#page-name" class="wikilink">Page Name</a>`. The link targets the heading ID generated by markdown-it-anchor.

The parser is configured with `html: true` so that raw HTML in the markdown (like the injected `<div>` anchors) passes through.

## Styling Architecture

All styling is in a single `style.css` file using **CSS custom properties** for theming.

### Theming

The colour palette is defined in `:root` (light mode) and overridden in `[data-theme="dark"]`. To change a colour, edit the variable — it applies everywhere.

Key variable groups:

| Prefix | What it controls |
|---|---|
| `--bg-*` | Backgrounds (page, sidebar, content, code blocks, tables, etc.) |
| `--text-*` | Text colours (primary, secondary, muted, links) |
| `--accent*` | The goblin-green accent colour and its variants |
| `--border*` | Border and divider colours |
| `--shadow-*` | Box shadow levels (sm, md, lg) |
| `--radius-*` | Border radius levels |

### Typography

- **Body text:** `Lora` (serif) — gives the "TTRPG manual" feel
- **UI elements** (sidebar, headers, tabs): `Inter` (sans-serif) — clean and modern
- **Code/monospace:** `JetBrains Mono`

Fonts are loaded from Google Fonts in the HTML templates.

### Layout

- Sidebar is `position: fixed`, `280px` wide, scrolls independently
- Main content has `margin-left: var(--sidebar-width)` to sit beside it
- Content is capped at `72ch` width for optimal reading line length
- On screens < 900px: sidebar hides off-screen and is toggled via hamburger menu

### Notable CSS Classes

| Class | Where it appears | Purpose |
|---|---|---|
| `.section-divider` | Injected by build.js | Invisible anchor for folder-level sidebar links |
| `.page-start` | Injected by build.js | Invisible anchor for file-level sidebar links; has `scroll-margin-top` so the header isn't hidden behind the top bar |
| `.wikilink` | Generated by wikilinks plugin | Styled differently from normal links (green background pill) |
| `.nav-link--active` | Added by scroll spy JS | Highlights the current section in the sidebar |

## Client-Side JavaScript

All JS is vanilla, embedded in `template.html`. There are four independent modules:

1. **Theme Toggle** — Reads/writes `data-theme` attribute on `<html>`. Persists choice to `localStorage` under key `gobbos-theme`.
2. **Sidebar Toggle** — Mobile hamburger menu. Adds/removes `.sidebar--open` class and shows a backdrop overlay.
3. **Sidebar Collapse** — The `▾`/`▸` buttons on folder sections. Toggles `display: none` on the child `<ul>` and updates `aria-expanded`.
4. **Scroll Spy** — On scroll, finds which `.page-start` anchor is nearest the top of the viewport and adds `.nav-link--active` to the corresponding sidebar link.

## How to Extend

### Adding a new environment

1. Create the directory in the repo root (e.g. `03_Supplements/`)
2. Add an entry to the `ENVIRONMENTS` array in `build.js`:
   ```js
   { key: 'supplements', dir: '03_Supplements', title: 'Supplements', subtitle: 'Optional expansions', icon: '📦' },
   ```
3. Rebuild. The landing page card and a new `supplements.html` will be generated automatically.

### Changing the visual theme

Edit the CSS custom properties in `style.css` under `:root` (light) and `[data-theme="dark"]`. No other files need to change.

### Adding assets (images, fonts)

1. Place files in a new `assets/` folder alongside `style.css`
2. In `build.js`, add a copy step after the CSS copy:
   ```js
   await fs.copy(path.join(__dirname, 'assets'), path.join(DIST_DIR, 'assets'));
   ```
3. Reference them in CSS or templates as `assets/filename.png`.

### Modifying the HTML template

Edit `template.html`. The placeholders are:

| Placeholder | Replaced with |
|---|---|
| `{{TITLE}}` | Page `<title>` (e.g. "STAGE Drafts — Gobbos") |
| `{{PAGE_TITLE}}` | Sidebar header text |
| `{{SUBTITLE}}` | Subtitle under the sidebar header |
| `{{NAV_CONTENT}}` | Generated sidebar HTML |
| `{{MAIN_CONTENT}}` | Rendered markdown HTML |
| `{{ENV_NAV}}` | Environment tab links in the top bar |

`{{TITLE}}` is replaced globally (used in `<title>` and potentially elsewhere). All others are replaced once.

### Adding a new markdown-it plugin

In `build.js`, inside `createMarkdownParser()`:
```js
const somePlugin = require('markdown-it-something');
md.use(somePlugin, { /* options */ });
```

## Cross-Platform Notes

This project is developed on Windows but designed to run on Linux (RPi5) without changes:

- All file paths use `path.join()` / `path.resolve()` — never hardcoded separators
- npm scripts use only Node.js commands (no PowerShell or bash-specific syntax)
- The output is pure static HTML/CSS/JS — serveable by any web server (nginx, Apache, Python `http.server`, etc.)

### Deploying to RPi5 / Linux

Copy the project, run `npm install` and `npm run build`, then serve `dist/` with your preferred web server. For example:

```bash
# nginx (copy dist contents to web root)
sudo cp -r dist/* /var/www/html/

# Or just serve directly
npx serve dist --port=3000
```
